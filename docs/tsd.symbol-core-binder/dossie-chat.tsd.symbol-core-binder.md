# 1. RESUMO DE TRANSFERENCIA

## 1.1 O que precisa sobreviver

```text
1. O usuario identificou que a IA tende a governar cedo demais quando deveria apenas escutar.
2. Essa observacao virou criterio de projeto: nem todo sinal pede interpretacao, estrutura ou decisao.
3. O problema tecnico real surgiu depois: resolver simbolos no core do TESSERUS.
4. O boot ja e tabelado, entao pode gerar um binder baseado em tabelas existentes.
5. O binder do boot nao deve se estender semanticamente para componentes que ainda nao possuem DocFIELD/domain pack.
6. A solucao proposta foi separar autoridade em camadas: BINDER_UNIVERSAL, BOOT_TABLE_BINDER, COMPONENT_BINDER_STUB, DOMAIN_PACK futuro, CROSSMAP e UNMAPPED.
7. O usuario propos uma traducao posicional/visual: nome do arquivo, marcador, nome da tabela e campos por posicao, associados a cores/emoji.
8. A cor/emoji foi aceita como legenda visual/META, mas bloqueada como simbolo canonico de core.
9. A identidade operacional dos campos deve vir de posicao/endereco; a semantica vem depois.
10. RM.DocFIELD deve receber binds, slots, tabelas, nomes internos, value streams e backlog UNMAPPED.
```

## 1.2 Frase para a proxima AI

```text
Continue a partir da ideia de que o TESSERUS deve resolver o problema dos simbolos no core por enderecamento posicional e autoridade gradual: o boot tabelado pode bindar o que existe, enquanto os demais componentes ficam em stubs/UNMAPPED ate DocFIELD/domain pack/crossmap validarem a semantica.
```

## 1.3 Nucleo irredutivel

```text
MECANISMO CENTRAL:
- Separar simbolizacao em tres niveis: posicao operacional, legenda visual e autoridade canonica.

GARGALO PRINCIPAL:
- Evitar que o binder do boot, por estar tabelado, colonize semanticamente componentes ainda nao materializados.

VARIAVEL GOVERNANTE:
- Grau de autoridade da origem do simbolo: tabela existente > DocFIELD > domain pack > crossmap > canone.

O QUE MUDARIA MINHA CABECA:
- Existencia de DocFIELD/domain packs completos para todos os componentes, com crossmap sem colisao e symbol map validado, permitindo promocao de parte dos stubs para simbolos canonicos.
```

# 2. DESCRICAO NAO TECNICA

```text
O QUE E:
- Um metodo para organizar simbolos internos do TESSERUS sem inventar significado antes da hora.

PARA QUE SERVE:
- Permite que tabelas ja existentes do boot sejam traduzidas para um binder, enquanto componentes ainda incompletos ficam reservados como stubs.

O QUE NAO E:
- Nao e fechamento.

- Nao e execucao real.
```
